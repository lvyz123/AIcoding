## 新增机制评估

**auto-disable full prefilter（新增，设计有问题）**

L3119-3124 新增了一个探测机制：前 512 对之后，若 full prefilter 拒绝率 < 2%，则对整个 bundle 禁用 full prefilter。

问题在于这个逻辑的方向和 prefilter 的作用是反的。**full prefilter 拒绝率低，说明候选之间相似度高、几何 pair 多，这正是最需要 full prefilter 来精确筛选的场景**，不应该禁用。反而拒绝率高的时候（候选之间差异显著），cheap filter 本身就已经够用，full prefilter 的额外价值有限。

当前逻辑等于"越需要 full prefilter 的场景越禁用它"，会导致 precision 下降。建议把这个 auto-disable 逻辑的判断条件反转，或者直接移除：

```python
# 当前（错误方向）
if reject_rate < 0.02:  # 拒绝率太低就禁用
    bundle["full_prefilter_disabled"] = True

# 改为（正确方向）：拒绝率极高说明 cheap filter 已足够，可以跳过 full
if reject_rate > 0.98:  # 拒绝率极高则 cheap 已经够用
    bundle["full_prefilter_disabled"] = True
```

## 一个残留的内存释放漏洞

ECC 路径中存在一条 target_chunk **未用 `finally` 包裹**的释放路径。当 `auto_true` 覆盖了所有 `overlap_indices`（即 `overlap_indices[~auto_true].size == 0`），代码会执行 `matched_chunks.append(...)` 后直接结束循环体，**跳过了 L3350 的 `_release_bundle_geometry_cache(bundle, target_chunk)` 调用**。

这个路径发生条件：`source_donut_area == 0` 或所有 `overlap_indices` 的 `target_donut_areas == 0`，即空 pattern 的匹配场景，在稀疏区域的 corner clip 里会触发。

ACC 和 ECC-tol_px=0 路径已经用 `try/finally` 包裹，但 ECC-tol_px>0 路径的 `for target_chunk` 循环没有。修复是给 ECC chunk 循环加 `finally`：

```python
for target_chunk in _chunk_indices_by_row_width(target_indices, packed_row_bytes * 4):
    try:
        # ... 原有 ECC 逻辑 ...
        self._release_bundle_geometry_cache(bundle, target_chunk)  # 删掉这行
    finally:
        self._release_bundle_geometry_cache(bundle, target_chunk)  # 改用 finally
```
